Props
Props — inputs from the parent
Props pass data from parent to child, like arguments to a function. The parent decides the values; the child receives and renders them.
Data flows one-way (parent → child). Inside the child, props are read-only. Need a change? The parent updates what it passes.
What can you pass as props?
  • Plain data: strings, numbers, booleans, objects, arrays
  • Functions the child can call (e.g. onClick={handleClick})
  • Other components: as a named prop or via children
function Welcome(props) {
  return <h1>Hello, {props.name}!</h1>;
}

// you can also destructure in the parameter
function Welcome({ name }) {
  return <h1>Hello, {name}!</h1>;
}

// parent chooses the name
<Welcome name="Aisha" />
Composition
The children prop
children is a special prop React sets automatically. Whatever JSX you place between a component's opening and closing tags (text, elements, other components) becomes children inside it.

This is very useful for composing UI: one component provides the structure, and the parent decides what "fills" it.
Define the shell once, reuse it anywhere
Build the wrapper component once, with its layout and structure, and let each parent pass in whatever content belongs there.
function Layout({ children }) {
  return (
    <>
      <header>…</header>
      <main>{children}</main>
      <footer>…</footer>
    </>
  );
}

<Layout>
  <h1>Dashboard</h1>
</Layout>
State
useState — data that changes over time
Props come from outside. State lives inside the component — and updating it triggers a re-render.
React hook for managing state — Function Signature:
const [
current value
,
setter function
] = useState (
initial value
);
import { useState } from 'react';

const [count, setCount] = useState(0);

// set a new value directly
setCount(count + 1);

// use previous value (safer)
setCount((c) => c + 1);
How it works
You pass an initial value to useState. It gives you back two things:
  • count — the current value
  • setCount — the setter function; call it to update and re-render
Rules of hooks
Call useState at the top level of your component. Not inside loops, conditions, or nested functions.
State
Updating objects and arrays
// wrong: mutating state directly
user.name = ’Lee’; // React won’t re-render

// correct: create a new value
setUser((u) => ({ ...u, name: ’Lee’ })); // new object
setItems((xs) => [...xs, item]); // new array
Why does this matter?
React detects changes by comparing the old and new value. If you mutate the existing object or array, the reference stays the same and React won’t notice anything changed.
The pattern
Objects: spread with { ...old, key: newValue }.
Arrays: use spread, .filter(), or .map() to return a new array.
Conditional UI
Show or hide based on props or state
Branch on props (parent decides) or state (this component decides). The patterns are the same either way.
Three patterns
if / else: cleanest for longer branches; return null to render nothing.

? : compact "either A or B" inline.

&&: show something only when the left side is true.
// if / else — return early
if (isLoading) return <p>Loading...</p>;

// ternary — either A or B
<button>
  {isSubmitting  ? 'Saving...'  : 'Save'}
</button>

// && — show only when true
{error && <p className="error">{error}</p>}
Lists
Rendering a list with .map()
When you have an array of data (cart items, messages, users…), you turn each item into JSX. React needs a key on each sibling in the list so it can track which row is which when the list changes.
return (
  <ul>
    {items.map((item) => (
      <li key={item.id}>
        {item.label}
      </li>
    ))}
  </ul>
);
Choosing a key
Prefer something stable and unique from your data (like an id). Using only the array index is a poor fit if rows can be reordered, added, or removed. React may update the wrong row.
Sharing state
Lifting state up
When two components need the same state, move it to their nearest shared parent and pass it down as props.
The parent passes the value and an updater function to each child. Data still flows one-way. The parent is the single source of truth.
function Parent() {
  const [text, setText] = useState('');
  return (
    <>
      <Editor
        value={text}
        onChange={setText}
      />
      <Preview text={text} />
    </>
  );
}

Key takeaways
● Props are inputs from the parent. Read them, don’t rewrite them. Need new data? The parent re-renders and passes different props (often after its own state changes).
● children is how you compose layouts: one component provides the frame, another fills the middle.
● useState is for values that change over time; updating state triggers a re-render. Treat objects and arrays as immutable when you update.
● Use .map() + stable keys for lists; move shared state up to the parent when two siblings need the same value.
