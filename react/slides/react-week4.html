Context
What is the Context API?
Built-in React feature to provide a value to a subtree of components. Consumers read it with useContext — no manual “prop drilling” through intermediate components.
Problems it helps with
  • Prop drilling — passing the same prop through many layers just to reach one leaf.
  • Cross-cutting state — things many distant components need (theme, language, signed-in user).
Keep perspective
Prop drilling is fine until it hurts maintainability. Also: composition (passing components as children) often avoids unnecessary context.
Not a full data layer
For large or complex global state, teams often reach for dedicated libraries — Context still has a sweet spot for app-wide settings and auth “session” objects.
Context
When should you use Context?
Use it when many components in different parts of the tree need the same data and passing props is noisy or brittle.
Theming
Light / dark mode, design tokens — one provider, many consumers.
Localization
Current locale, translators, date/number formatting.
Auth / current user
Who is signed in — without passing user through every layout.
Guideline
Prefer local state and props first. Introduce Context when the dependency graph gets wide or deep and the prop names repeat everywhere.
Context
createContext and the default value
import {createContext, useState} from 'react';

export const UserSettingsContext = createContext(null);
// default (e.g. null) is used if there is no matching Provider above
Default value
Shown only when a component reads context without an ancestor Provider. Handy for tests and spotting mistakes — production trees should almost always wrap with a Provider.
Split contexts
Avoid one giant context that changes often — unrelated consumers would re-render together. Separate “theme” from “locale”, for example.
Context
Provider and the value prop
Wrap part of your tree with YourContext.Provider . The value prop is what subscribers see. You can lift state next to the Provider or build a tiny wrapper component — both are fine.
export function AppPage() {
  const [userLanguage, setUserLanguage] = useState('en');

  return (
    <UserSettingsContext.Provider
      value={{ language: userLanguage, setUserLanguage: setUserLanguage }}
    >
      <App />
    </UserSettingsContext.Provider>
  );
}
Context
Read with useContext — the recipe
import {useContext} from 'react';
import {UserSettingsContext} from './AppPage';

export function Profile() {
  const { language, setUserLanguage } = useContext(UserSettingsContext);

  return (
    <section>
      <h2>My language</h2>
      <select
        value={language}
        onChange={(e) => setUserLanguage(e.target.value)}
      >…</select>
    </section>
  );
}
The Context recipe
  1. createContext (optional default)
  2. Wrap the app (or a subtree) in a Provider
  3. Pass the live data via value
  4. Read with useContext
Shape of `value`
Match what you pass in value with what you destructure in useContext — typos here are a common bug.
Routing
React Router
For apps that are not using a framework like Next.js, the React docs point to react-router for URL-based navigation and rendering different screens.
Three modes (high level)
Framework, Data, and Declarative. For this course we focus on Declarative routing — less powerful than the others, but the easiest mental model when you are learning.
Why not plain <a href>?
Full page reloads lose client state. Router-driven links swap components on the client while keeping one SPA shell — faster UX.
# npm (or yarn / pnpm)
npm install react-router

Docs: Getting Started shows BrowserRouter + Routes + Route.
Routing
Setup: BrowserRouter wraps the tree
Refactor entry (e.g. main.jsx ) so the root renders BrowserRouter , Routes , and each Route . Your existing App can be the component for path="/" .
import {StrictMode} from 'react';
import {createRoot} from 'react-dom/client';
import {BrowserRouter, Routes, Route} from 'react-router';
import App from './App.jsx';

createRoot(document.getElementById('root')).render(
  <StrictMode>
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<App />} />
      </Routes>
    </BrowserRouter>
  </StrictMode>,
);
Routing
More routes & navigation with NavLink
<Routes>
  <Route path="/" element={<HomePage />} />
  <Route path="/about" element={<AboutPage />} />
  <Route path="/contact" element={<ContactPage />} />
</Routes>

<nav>
  <NavLink to="/">Home</NavLink>
  <NavLink to="/about">About</NavLink>
</nav>
NavLink vs Link
Both render anchors with client-side navigation. NavLink adds optional styling when the URL matches — useful for highlighting the active tab in a navbar.
Experiment in class
Add two or three trivial page components — change the URL in the address bar — show that the correct component renders without a full refresh.
UI ecosystems
Component & UI libraries
What they are
Ready-made building blocks — buttons, dialogs, layouts. Some are heavily styled (e.g. Material UI); others are mostly unstyled “primitives” (e.g. Radix-style libraries).
Benefits
Consistency across the product, speed versus hand-rolling everything, and usually better accessibility defaults than bespoke one-offs.
Challenges
Learning curve — every library names props differently. Lock-in — mixing several full systems rarely pays off; migration can mean a big refactor.
For exercises

Key takeaways
● Context
Use createContext , wrap with Provider and value , read with useContext . Prefer props and composition until shared data is truly global or deeply nested.
● React Router (declarative)
BrowserRouter at the root, Routes lists Route entries with path and element . Use NavLink or Link for in-app navigation without full reloads.
● UI libraries
Trade learning cost and lock-in for speed, consistency, and a11y. Many expose their own Provider patterns — good bridge from this week’s theory to real-world code.
